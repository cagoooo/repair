# 未來開發藍圖與建議 (Future Roadmap & Suggestions)

基於目前的系統架構 (React + Firestore + LINE Notify)，以下提供關於功能擴充、體驗優化與技術升級的詳細建議，作為未來維護與改良的參考。

## 🚀 短期目標 (Short-term Goals) - 智慧化與管理升級

### 1. 耗材庫存管理 (Inventory Management)
- **概念**: 報修通常伴隨著耗材使用 (例如更換燈管、滑鼠、網路線)。
- **實作**:
  - 建立簡易庫存系統 (物品名稱, 剩餘數量)。
  - 管理員於後台點擊「結案」時，可勾選本次消耗的物件與數量 (例如「更換燈管 x2」)，系統自動扣除庫存。
  - **加值功能**: 當庫存低於設定的水位 (如低於 5 個) 時，自動發送 LINE 通知提醒事務組/資訊組補貨。

### 2. AI 智慧客服 / 預判報修 (Smart Assistant)
- **概念**: 在填寫報修單時，系統即時給予初步排除障礙的建議，並協助自動分類。
- **實作**:
  - 透過簡易的關鍵字比對或是串接 Gemini API 分析報修描述 (例如輸入「無法開機」時，系統跳出提示框：「請先檢查電源線是否鬆脫」)。
  - 若能引導老師自行排除問題，不僅能提升效率，還能大幅減少不必要的人工維修負擔。

### 3. 主題系統與深色模式 (Dark/Light Mode)
- **現況**: 目前 UI 主要採用深色系的 Aurora Dark 風格。
- **建議**:
  - 新增「淺色/深色」快速切換開關，讓白天如果在室外強光下使用時，能有更清楚的高對比淺色介面。
  - 透過 CSS 變數 (`--bg-gradient`, `--text-primary` 等) 進行主題的全域抽換。

## 🌟 中期目標 (Mid-term Goals) - 互動優化與服務擴展

### 4. 掃碼溯源自動帶出報修 (QR Code Integration)
- **概念**: 利用現有的資產標籤，加速報修輸入流程。
- **實作**:
  - 於報修首頁或 PWA App 中實作「掃描條碼 / QR Code」按鈕。
  - 利用 Web Auth API / 瀏覽器鏡頭 API 掃描設備上的資產條碼或教室門口的專屬 QR Code。
  - 掃描成功後，系統直接跳轉至報修表單，並「自動填寫」地點或設備名稱。
  - **優點**: 幾乎可以消滅報修時地點填寫錯誤或描述不清的問題。

### 5. 即時對話與溝通模組 (Real-time Messaging)
- **現況**: 目前透過增加 `comments` (處理備註) 進行溝通，但對報修者來說多半是單向的。
- **建議**:
  - 允許報修者也能在自己的單據下補充留言 (例如上傳新照片：「目前水管口變成噴水了」)。
  - 管理員回覆後，系統能透過機制 (如 Service Worker Push Notification) 推送通知給原始報修者，讓溝通變成雙向互動。

### 6. 優化統計圖表與自動化報表 (Automated Reports)
- **現況**: 目前有簡單的儀表板與熱力圖。
- **建議**:
  - 增加按月或按學期「自動結算匯出」 PDF 報表的功能。
  - 將「修繕效率」量化 (例如 MTTR 趨勢圖)，並分析每月的維修重點，作為次學期預算編列的真實數據依據。

## 🛠️ 長期目標 (Long-term Goals) - 架構升級與安全性

### 7. 後端邏輯雲端化 (Firebase Cloud Functions)
- **現況**: 目前像是 LINE Notify 呼叫或是狀態轉換的邏輯多在前端或 GAS 處理。
- **建議**:
  - 將敏感邏輯 (寄送通知、自動扣庫存核算) 全面移至 Firebase Cloud Functions 撰寫。
  - **優點**: 避免將通知金鑰或流程暴露在前端，系統更安全，修改商業邏輯也不再需要使用者更新快取頁面。

### 8. 權限角色精細劃分 (RBAC - Role-Based Access Control)
- **現況**: 系統僅二分為 單純使用者 / 管理員。
- **建議**:
  - 將管理員再往下劃分，並賦予不同的資料存取範圍。
    - **系統管理員**: 擁有所有設定與增刪改管理的最高權限。
    - **資訊組長**: 只能看到且處理「資訊 IT 類別」的報修，並管理資訊庫存。
    - **事務組長**: 只能看到且處理「一般修繕類別」的報修，並管理一般庫存。
    - **維修志工 / 工讀生**: 僅具備「更新進度為處理中」及「留言」的權限，不能刪除或直接結案。

### 9. 導入 TypeScript 與單元測試
- **建議**: 隨著專案模組數量擴增，原本單純的 JavaScript 可能帶來型別上的維護風險。
  - 逐步將核心邏輯替換為 `.tsx` / `.ts`，為資料如 `Repair`、`Room` 訂定 Interface，提前防堵 Runtime 錯誤。
  - 對最核心的工具函式撰寫 Jest 自動化單元測試，確保後續擴增功能時核心依然穩固。

| :--- | :--- | :--- | :--- |
| 🔥 **高度推薦** | **導入耗材庫存紀錄管理** | 完美對接維修結案的後續流程，將報修與物料庫存結合，具極高真實行政價值。 | 中 |
| 🔥 **高度推薦** | **掃描條碼自動帶入報修** | 徹底將報修體驗降到「0門檻」，讓資訊化體現在極簡操作上。 | 中高 |
| 🟡 **中等排序** | **報修者與維修者雙向留言** | 處理特殊懸案或需要釐清細節時非常實用。 | 中 |
| 🟡 **中等排序** | **AI 智慧初步排除建議** | 第一時間攔截並減少極易解決的無效報修任務。 | 中高 |
| 🟢 **基礎建設** | **後端與權限系統精細化** | 迎合校園規模擴大、分工變細的長期必備架構。 | 高 |
