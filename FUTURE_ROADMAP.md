# 未來開發藍圖與建議 (Future Roadmap & Suggestions)

基於目前的系統架構 (React + Firestore + LINE Notify)，以下提供關於功能擴充、體驗優化與技術升級的詳細建議。

## 🚀 短期目標 (Short-term Goals) - 提升易用性

### 1. 數據分析儀表板 (Analytics Dashboard) 2.0
- **現況**: 目前已有基礎的圓餅圖與長條圖。
- **建議**:
  - **維修效率分析**: 計算「平均完修時間」(MTTR)，例如「鍵盤維修平均需 2 天」。
  - **熱點分析**: 結合地圖，以熱力圖 (Heatmap) 顯示常報修的教室區域。
  - **個人貢獻**: 顯示「報修王」或「維修超人」排行榜（遊戲化 Gamification）。

### 2. PWA 安裝優化 (App-like Experience)
- **現況**: 已有 `manifest.json` 與 Service Worker。
- **建議**:
  - **自訂安裝提示**: 在 iOS/Android 上跳出「加入主畫面」的教學指引。
  - **推播通知 (Web Push)**: 除了 LINE Notify，增加瀏覽器原生的 Web Push Notification，讓使用者在沒開 App 時也能收到進度通知 (需配合 FCM)。

### 3. 深色模式 (Dark Mode) 切換
- **現況**: 目前 UI 以深色為主 (Aurora Dark)。
- **建議**:
  - 新增「淺色/深色」切換開關，適應白天強光下的操作場景。
  - 將 CSS 變數標準化，方便一鍵切換主題色系。

## 🌟 中期目標 (Mid-term Goals) - 智慧化與自動化

### 4. AI 智慧客服 / 預判報修 (Smart Assistant)
- **概念**: 在填寫報修單時，AI 自動建議解決方案。
- **實作**:
  - 當使用者輸入「網路不通」時，系統跳出提示：「請先嘗試重啟分享器 (附圖)」。
  - 利用 OpenAI API 或 Gemini API 分析報修描述，自動歸類並判斷優先級。

### 5. 耗材庫存管理 (Inventory Management)
- **概念**: 報修通常伴隨著耗材使用 (如滑鼠、線材)。
- **實作**:
  - 建立簡易庫存系統 (Item, Quantity)。
  - 管理員結案時，可勾選「更換設備：滑鼠 x1」，系統自動扣庫存。
  - 庫存低於水位時發送 LINE 通知提醒補貨。

### 6. 掃碼報修溯源 (QR Code Traceability)
- **概念**: 讓每台設備都有身分證。
- **實作**:
  - 為每間教室或重要設備產下 QR Code。
  - 掃描即自動帶入「教室」、「設備名稱」，不僅省去輸入，還能精確追蹤該設備的維修歷史。

## 🛠️ 長期目標 (Long-term Goals) - 架構升級

### 7. 後端邏輯雲端化 (Cloud Functions)
- **現況**: 目前 `isAdmin`、通知發送等邏輯多在前端或透過 GAS。
- **建議**:
  - 將敏感邏輯 (如刪除權限檢查、LINE Token 存取) 移至 Firebase Cloud Functions。
  - **好處**: 安全性大幅提升，前端看不到任何 Secret，也不用依賴 localStorage 判斷身分。

### 8. 權限分級系統 (RBAC)
- **現況**: 只有 Admin / User 二分法。
- **建議**:
  - 實作角色權限控制 (Role-Based Access Control)。
  - **角色**:
    - **超級管理員**: 所有權限。
    - **事務組長**: 僅能管理「冷氣、水電」類報修。
    - **資訊組長**: 僅能管理「電腦、網路」類報修。
    - **維修工讀生**: 僅能更新狀態，不能刪除。

### 9. 導入 TypeScript
- **建議**: 隨著專案變大，JavaScript 的弱型別可能導致難以維護。
- **行動**: 逐步將 `.jsx` 遷移至 `.tsx`，增加介面 (Interface) 定義，減少 Runtime Error。
